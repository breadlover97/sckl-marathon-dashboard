How the backend works

Google Sheets is the editable training plan, Strava is the actual running log, GitHub Actions performs the private sync, Cloudflare Worker handles run-note writes, and GitHub Pages serves the public dashboard.

Information Flow

1You edit the marathon plan in Google Sheets. The sheet stays the source of truth for planned weeks, dates, daily mileage, notes, and weekly summaries.
2You record runs normally through Strava. The dashboard only needs Strava API access; it does not need Strava data typed into the sheet.
3GitHub Actions runs the sync workflow on schedule, on code changes, or when you press the Sync data button and manually run the workflow.
4The workflow reads private credentials from GitHub secrets, fetches Google Sheets and Strava data, then writes generated JSON files into the Pages artifact.
5GitHub Pages hosts the static site. The browser loads data/training-plan.json and data/strava-activities.json to render the dashboard.
6Run notes go the other direction: after passcode auth, the browser sends a note to the Cloudflare Worker, and the Worker writes it to the Run Notes tab in Google Sheets. The Worker creates that tab on first use if it does not exist yet.
7Supplement checks use the same Worker pattern. The browser sends the date and supplement state to /supplements, then the Worker writes it to the Supplements tab.

What Each Service Does

Google Sheets stores the planned marathon build. You can adjust daily estimated mileage, session notes, weekly summaries, and phase details there.

Strava API provides actual runs: distance, time, pace, elevation, heart rate and cadence when Strava includes them, activity names, dates, and Strava links.

GitHub Actions is the private sync worker. It refreshes Strava tokens, reads the Google service account secret, fetches both data sources, and prepares the Pages deployment.

Cloudflare Worker accepts authenticated run-note and supplement writes from the static website and updates the Google Sheet without exposing Google credentials.

Browser localStorage caches supplement check history and the dashboard passcode locally. Google Sheets becomes the cross-device source of truth after the Worker is deployed.

GitHub Pages is the public static host. It has no server-side code and no API secrets in frontend JavaScript.

Workflow Responsibilities

Google Sheets sync

scripts/fetch_google_sheet.py reads the configured sheet range, validates expected columns, parses each week, and writes data/training-plan.json.

Strava sync

scripts/fetch_strava.py refreshes the access token with your refresh token, fetches recent activities, keeps running activities, and writes public training metrics.

Pages deployment

deploy-pages.yml copies HTML, CSS, JavaScript, and generated JSON into the Pages artifact. The hosted site is just static files.

Manual refresh

The Sync data button opens the GitHub Actions workflow page. From there, run the workflow manually to fetch the latest sheet and Strava data.

Run notes

The Cloudflare Worker reads and writes notes in the Run Notes tab, keyed by Strava activity ID, after you enter the private passcode in the browser. The tab is created automatically the first time notes are loaded or saved.

Supplements

The same Worker reads and writes the Supplements tab, keyed by date, so supplement checks stay consistent across devices.

Pace Range Derivation

The pace table is a practical coaching estimate, not a lab result. It starts from recent performance anchors, converts those into equivalent training intensities, then widens the final ranges for Singapore heat, humidity, hills, and day-to-day fatigue.

Performance anchor

The baseline estimate uses a 37:00 tropical 10K, cross-checked against a 1:19 half marathon and historical 2:45 cool-weather marathon fitness. This points roughly to a VDOT-style fitness level around 57.

Marathon target

A 2:50 marathon requires about 4:02/km in neutral conditions. For SCKL conditions, the working marathon-effort range is softened to roughly 4:10-4:20/km so effort stays sustainable in heat.

Training zones

Threshold, tempo, interval, and repetition ranges are built around common race-equivalent training intensities, then expressed as ranges rather than exact single paces.

Adjustment rule

If sleep, shin soreness, humidity, or hills raise the effort, the intended stimulus wins over the number on the watch. Easy days should stay easy enough to support the next key session.

Secrets And Runtime Config

GitHub repository secrets

GOOGLE_SERVICE_ACCOUNT_JSON, STRAVA_CLIENT_ID, STRAVA_CLIENT_SECRET, and STRAVA_REFRESH_TOKEN. These stay private in GitHub Actions.

GitHub repository variables

GOOGLE_SHEET_ID points to your training sheet. GOOGLE_SHEET_RANGE is currently A:AG so the generated weekly summary column is included.

Frontend config

The frontend only knows public file paths such as data/training-plan.json. It never receives Google or Strava secrets.

Local development

.env can hold local copies of the same values for testing scripts, while .env.example documents the required names without real credentials.

Cloudflare Worker secrets

GOOGLE_SERVICE_ACCOUNT_JSON lets the Worker write to Sheets. RUN_NOTES_TOKEN is the private passcode for loading and saving notes and supplement checks.

Data Contract

data/training-plan.json contains the plan from Google Sheets: week number, week start date, phase, weekly mileage, daily sessions, long run target, weekly summary, and notes.

data/strava-activities.json contains the generated actual training data used by the dashboard: athlete profile, sync timestamp, run distance, moving time, elapsed time, elevation, heart rate, cadence, activity names, dates, and Strava links.

The Worker serves supplement checks from the Supplements tab as date-keyed rows: date, protein shake, omega 3, vitamin D, and updated timestamp.

Privacy And Limits

The website is public, so anything written into generated JSON can be viewed by anyone with the site link. API secrets are private, but published activity metrics are public.

Run notes and supplement checks are different: they are served by the Cloudflare Worker and require the private passcode before the browser can load or save them.

Supplement checks are cached in this browser's localStorage, but the Supplements tab is the cross-device source of truth once the Worker is configured.

If a passcode, Strava token, or Google service account key is ever shared outside the secret stores, rotate it and update GitHub Actions or Cloudflare Worker secrets.

The dashboard is a training visibility tool, not a coaching or medical system. Risk flags and progress views should guide decisions, not replace judgement around injury, sleep, heat, or recovery.
